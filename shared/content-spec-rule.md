# Content spec rule

Before executing any substantial content work (500+ words, document generation,
multi-section drafting, translation, or content expansion), compose a Content Spec
first and get user approval before proceeding.

## Content Spec format

```
## Content Spec: [title]

**Type**: [report / article / guide / proposal / memo / other]
**Length**: [target word count or page count]
**Audience**: [who reads this]
**Tone**: [formal / conversational / technical / other]

### Structure
1. [Section] - [one-line description of what goes here]
2. [Section] - [description]
3. [Section] - [description]

### Key points to cover
- [point]
- [point]

### Exclusions (what NOT to cover)
- [exclusion]

### Output format
[markdown / docx / pdf / pptx]
```

Wait for explicit approval ("go", "approved", "looks good") before writing.

If the user says "skip spec" or "just write it", proceed without the spec.

For the full content-spec workflow with requirements ledger and iteration
tracking, use the content-spec skill from the claude-skills repo.
